RAG는 Retrieval-Augmented Generation의 약자입니다.
사용자의 질문과 관련된 문서를 먼저 검색한 뒤,
검색된 내용을 LLM에게 전달하여 답변을 생성하는 방식입니다.

RAG를 사용하면 챗봇이 내용을 외우는 것이 아니라 찾아서 답변합니다.
LLM 단독으로 답변할 때보다 근거 기반 답변이 가능하고,
학습 데이터에 없는 최신 문서나 개인 문서에 대해서도 답할 수 있습니다.

RAG의 기본 흐름은 다음과 같습니다.
첫째, 문서를 작은 청크로 나눕니다.
둘째, 각 청크를 임베딩 모델로 숫자 벡터로 변환합니다.
셋째, 벡터를 벡터DB에 저장합니다.
넷째, 사용자 질문이 들어오면 질문도 임베딩하여 비슷한 청크를 검색합니다.
다섯째, 검색된 청크를 프롬프트에 넣어 LLM이 답변을 생성합니다.

임베딩은 문장을 고정 길이의 숫자 벡터로 바꾸는 과정입니다.
의미가 비슷한 문장은 벡터 공간에서 서로 가까운 위치에 놓입니다.
그래서 질문 벡터와 가까운 문서 벡터를 찾으면 관련 문서를 검색할 수 있습니다.

벡터DB는 임베딩 벡터를 저장하고 유사도 검색을 빠르게 해주는 데이터베이스입니다.
대표적으로 ChromaDB와 FAISS가 있습니다.
ChromaDB는 설치가 쉽고 로컬 파일로 저장할 수 있어 소규모 프로젝트에 적합합니다.

청크는 문서를 검색 단위로 자른 작은 덩어리입니다.
문서를 통째로 임베딩하면 여러 주제가 섞여 검색 정확도가 떨어지기 때문에,
CHUNK_SIZE 크기로 자르고 CHUNK_OVERLAP만큼 앞뒤 문맥을 겹쳐서 나눕니다.
CHUNK_OVERLAP을 두는 이유는 청크 경계에서 문장이 끊겨 의미가 사라지는 것을 막기 위해서입니다.

TOP_K는 검색 결과를 몇 개 가져올지 정하는 값입니다.
TOP_K가 너무 작으면 필요한 정보를 놓치고,
너무 크면 관련 없는 문서가 섞여 답변 품질이 떨어질 수 있습니다.
소규모 프로젝트에서는 TOP_K를 3 정도로 시작하는 것이 일반적입니다.
