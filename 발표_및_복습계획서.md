## 2. 발표 스크립트 (이렇게 설명하면 됩니다)

### 2-1. 전체 구조

> "두 개의 독립된 FastAPI 서비스로 구성했습니다.
> 하나는 로그인형 문서 기반 RAG 챗봇(Leon's ChatBot)이고,
> 다른 하나는 카카오모빌리티 API를 연동한 배송 서비스(MoveOps)입니다.
> 두 서비스를 하나로 합치기보다, 각 서비스 목적에 맞는 전용 AI를 따로 설계하는 방향으로 진행했습니다."

**왜 합치지 않았나?** → 문서 검색(RAG)과 배송 주문(함수 호출/API 실행)은 완전히 다른 종류의 작업이라서,
같은 파이프라인을 재사용할 수 없었고 실제로 그래프 로직 자체를 새로 설계해야 했습니다.

### 2-2. 사용한 모델 / 기술 스택

| 구성 요소 | 사용 기술 | 역할 |
|---|---|---|
| 주 LLM | **Claude Haiku 4.5** (`claude-haiku-4-5-20251001`) | 의도 분류, 정보 추출, 대화 응답 생성 |
| 폴백 LLM | **Gemini 2.5 Flash Lite** | Claude API 장애 시 자동 전환 |
| 오케스트레이션 | **LangGraph** (StateGraph) | 대화 흐름을 노드 그래프로 관리 (자기교정형 Agent) |
| 문서 검색(RAG) | ChromaDB + FAISS + BM25 하이브리드 | 문서 청크 검색 (RAG 챗봇 쪽) |
| 임베딩 모델 | sentence-transformers (다국어) | 문서를 벡터로 변환 |
| 직접 학습 모델 | 자체 제작 소형 Transformer (`chatbot/model.py`) | "내 로컬 채팅" 모드에서 사용하는 자체 학습 모델 |
| 배송 API | 카카오모빌리티 퀵·도보배송 **Sandbox** API | 실제 견적/주문/취소 처리 |
| 지오코딩 | 카카오 로컬(Local) REST API | 주소/장소명 → 위도경도 좌표 변환 |

**"어떤 모델 썼어요?"에 대한 답변:**
> "대화 응답은 Anthropic의 Claude Haiku 4.5를 기본으로 쓰고, API 장애 시 Google Gemini 2.5 Flash Lite로
> 자동 폴백하도록 만들었습니다. 두 모델 다 매번 새로 파인튜닝하는 게 아니라, 프롬프트 엔지니어링으로
> 의도 분류·정보 추출·응답 생성을 각각 지시해서 쓰는 방식입니다. 이 라우터(LLMRouter)는 RAG 챗봇과
> 배송 에이전트가 코드를 공유해서 씁니다."

### 2-3. RAG(Retriever)가 뭔지 설명하는 법

> "RAG는 Retrieval-Augmented Generation의 줄임말로, LLM이 원래 모르는 내용(제 개인 문서 등)을
> 답변에 반영하게 만드는 방법입니다. 순서는:
> 1) 문서를 미리 잘게 쪼개서(청킹) 벡터로 변환해 저장해두고 (ChromaDB/FAISS/BM25)
> 2) 사용자 질문이 들어오면 그 질문과 가장 비슷한 문서 조각을 검색해서 (Retriever)
> 3) 검색된 문서를 LLM에 '이 내용을 참고해서 답해줘'라고 같이 넘겨서 답변을 생성합니다 (Generation)
> 이렇게 하면 LLM이 모르는 최신/개인 정보도 답할 수 있고, 문서에 없으면 '모른다'고 정직하게 답하게
> 설계했습니다 (환각 방지)."

**리트리버가 여러 개인 이유(하이브리드 검색)**: 벡터 검색(ChromaDB/FAISS)은 의미가 비슷한 문장을 잘 찾고,
BM25는 정확한 단어/키워드 매칭을 잘 찾기 때문에 세 방식을 가중합으로 섞어서(하이브리드) 검색 정확도를 높였습니다.

### 2-4. 이번에 만든 배송 대화형 에이전트 (핵심 발표 포인트)

**동작 흐름**: "판교역에서 정자역으로 서류 하나 퀵으로 보내주세요" 같은 자연어 요청을 받으면

1. **의도 분류**: 정보 제공/수정/확정/취소/상태조회/잡담 중 뭔지 LLM이 분류
2. **정보 추출(슬롯 채우기)**: 메시지에서 출발지·도착지·물품·연락처 등을 JSON으로 추출해 대화 세션에 누적
3. **지오코딩**: "판교역"같은 장소명을 카카오 로컬 API로 실제 위도/경도 좌표로 변환 (주소 검색 실패 시 키워드/장소 검색으로 재시도)
4. **완결성 검증**: 기존에 이미 만들어둔 Pydantic 데이터 모델(`CreateDeliveryRequest`)의 검증 규칙을 그대로 재사용해서 "무엇이 빠졌는지" 자동으로 판단 → 부족하면 한국어로 되물음
5. **견적 조회**: 카카오모빌리티 Sandbox `price` API를 실제로 호출해 가격을 보여줌
6. **확정 전 안전장치**: 사용자가 명시적으로 "네/진행해줘"라고 확인해야만 실제 주문 API를 호출하도록 막아둠. 가격 확인 후 조건을 바꾸면 예전 가격으로 주문되지 않도록 자동 재견적
7. **주문 생성/상태조회/취소**: 실제 Sandbox 주문이 생성되고, "상태 확인해줘"/"주문 취소해줘"도 대화로 처리 가능

**실제로 검증한 것**: 이 흐름을 curl로 직접 호출해서 실제 Sandbox 주문(`orderId`, `MATCHING` 상태)이 생성되는 것까지 확인했습니다.

### 2-5. UI

MoveOps 웹 화면 좌측 하단에 플로팅 챗봇 위젯을 추가했고, **"AI 채팅"과 "내 로컬 채팅" 두 모드를 토글**로
전환할 수 있게 만들었습니다. AI 채팅은 위에서 설명한 배송 에이전트(Claude→Gemini)이고,
내 로컬 채팅은 제가 직접 학습시킨 소형 모델(질문-답변 쌍으로 학습, `chatbot/train.py`)이 답하는 모드입니다.

## 3. 예상 질문 Q&A

**Q. 어떤 모델을 썼나요?**
> Claude Haiku 4.5를 기본으로, 장애 시 Gemini 2.5 Flash Lite로 자동 폴백합니다. 별도 파인튜닝 없이 프롬프트로 역할(의도분류/정보추출/응답생성)을 나눠 씁니다.

**Q. RAG랑 이번 배송 에이전트는 뭐가 다른가요?**
> RAG는 "문서를 검색해서 답변에 근거로 쓰는" 구조이고, 배송 에이전트는 "대화에서 정보를 추출해 실제 외부 API(카카오모빌리티)를 호출하는" 구조입니다. 둘 다 LangGraph로 상태 흐름을 관리하지만 그래프 노드 내용은 완전히 다릅니다.

**Q. 왜 LangGraph를 썼나요?**
> 대화가 단순 1턴 질문-답변이 아니라, "정보 부족 → 되묻기 → 재검색/재추출 → 확정" 같은 반복·분기가 있는 흐름이라, 이를 명시적인 상태 그래프(노드+조건부 엣지)로 관리하는 게 조건문 범벅보다 명확했습니다.

**Q. 실제로 결제/주문이 잘못 나가는 걸 어떻게 막았나요?**
> 세 가지 안전장치를 넣었습니다. (1) 슬롯 추출만으로는 절대 주문 생성 API를 호출하지 않고, 사용자가 명시적으로 확정해야만 진행 (2) 견적을 보여준 뒤 조건이 바뀌면 예전 가격 그대로 주문되지 않도록 해시 비교로 재견적 (3) 동일 견적에 대한 중복 확정("네" 두 번 등)은 결정론적 주문 ID로 자동 중복 방지.

**Q. 지금 실제 서비스로 낼 수 있나요?**
> 아직 카카오모빌리티 API가 Sandbox(테스트) 단계라 실사용은 카카오 측 프로덕션 승인/정산 계약이 필요합니다. 지금은 Sandbox 환경에서 전체 흐름이 정상 동작하는 것까지 검증한 단계입니다.

**Q. 로컬 모델이랑 AI 모드는 뭐가 다른가요?**
> 로컬 모델은 제가 직접 학습시킨 소형 Transformer로, 배운 QA 쌍 안에서만 답할 수 있는 제한된 모델이고, AI 모드는 실제 Claude/Gemini를 호출해 배송 주문까지 처리할 수 있는 모드입니다. 토글로 언제든 바꿀 수 있게 UI에 넣었습니다.

## 4. 다음 주 미팅까지 복습계획서

- MoveOps 페이지에 좌하단 플로팅 챗봇 위젯 UI 추가, "AI 채팅 / 내 로컬 채팅" 토글 구현
- 실제 Sandbox 환경에서 대화 → 견적 → 확정 → 주문 생성 → 상태조회 → 취소 전 과정 동작 확인

### 다음 미팅 전까지 복습할 항목
1. **RAG/리트리버 개념 재정리**: 벡터 검색 vs BM25 키워드 검색의 차이, 하이브리드 스코어링 원리 (`chatbot/indexes.py` 코드 다시 읽고 직접 설명해보기)
2. **LangGraph 상태 그래프 구조**: `AgentState` (TypedDict), 노드/조건부 엣지 개념을 그림으로 그려보고 두 에이전트(RAG용/배송용)의 그래프를 비교 설명할 수 있도록 연습
3. **Pydantic 검증 재사용 원리**: 왜 "완결성 체크"를 직접 if문으로 안 짜고 기존 데이터 모델의 validator를 재사용했는지 (`CreateDeliveryRequest`) 다시 설명해보기
4. **동시성/비동기 이슈**: `httpx.AsyncClient`가 이벤트 루프에 묶이는 이유, `asyncio.to_thread`로 동기 LLM 호출을 감싼 이유 — 왜 이게 중요한지 한 문장으로 설명할 수 있게 정리
5. **한계점 파악**: Sandbox 단계라는 점, 다중 경유지 미지원, 지오코딩 실패 시 동작, LLM 오분류 가능성 등 — 강사님이 "한계가 뭐예요?"라고 물으면 바로 답할 수 있게 미리 정리
